00030 // assert() is commonly used to check assumptions made by the programmer, and
00031 // will generate a run-time error when the condition passed as a parameter
00032 // fails. The confirm() macro works like a better version of assert() with a
00033 // slightly different release-mode behavior.  This is best demonstrated with an
00034 // example:
00035 /*
00036 
00037 bool isEqualToZero(int* ptr) {
00038   confirm (ptr != NULL) else return false;
00039   return (*ptr) == 0;
00040 }
00041 
00042 */
00043 // In debug mode, this compiles to the equivalent of:
00044 // 
00045 //  bool isEqualToZero(int* ptr) {
00046 //    if (ptr != NULL) {
00047 //      static bool ignore_always = false;
00048 //      if (!ignore_always) ignore_always = assert_dialog("ptr != NULL");
00049 //      return false;
00050 //    }
00051 //    return (*ptr) == 0;
00052 //  }
00053 // 
00054 //  In release mode, this code becomes:
00055 // 
00056 //  bool isEqualToZero(int* ptr) {
00057 //    if (ptr == NULL) return false;
00058 //    return (*ptr) == 0;
00059 //  }
00060 // 
00061 // 
00062 // If the above code were written with assert(ptr != NULL), the release-mode
00063 // version would crash if the pointer were ever invalid since assert compiles
00064 // to (noop), and the program would dereference null.  In many cases, such as
00065 // the above example, there is some default behavior that can be executed so
00066 // that the program doesn't have to crash in the case of an error.
00067 // Additionally, when you want to use an assert() you can confirm() without the
00068 // else clause to take advantage of its always-ignore feature.  confirm() can
00069 // also be set to a logging release-mode, which behaves exactly like release-
00070 // mode but will create a debug console message each time a confirm() is hit.
00071 //
00072 // There is one more trick to using confirm().  Sometimes you want to make sure
00073 // a variable does have a certain value, then do something.  This is done in
00074 // the following example:
00075 /*
00076 
00077 void setPointerValue(int* ptr, int value) {
00078  confirm (ptr != NULL) then {
00079    *ptr = value;
00080  }
00081 }
00082 
00083 */
00084 // The use of "then" can make your code very clear.  This compiles in release
00085 // and debug mode to what you would expect:  both will execute then { ... }
00086 // only when ptr != NULL.  If ptr == NULL in debug mode, a run-time error will
00087 // be raised.
00088 // 
00089 // Like with assert(), do not use confirm() to check for erroneous user input,
00090 // file data, or other information that could be invalid as part of the
00091 // program's normal run-time behavior.  It should be used exclusively to check
00092 // program logic.
00093 // 
00094 // Defend your code!  Use confirm()!
00095 
00096

00098 //  - to use the default confirm() failure handler, add confirm.cpp to your project
00099 //  - confirm(), by default, will display messages in debug mode but not release mode
